# AUB Self-Dogfood Policy

AUB should use its own safety workflow for UI changes to the editor, workspace
drawer, evidence review panel, and GitHub Pages demo.

## Required For AUB UI PRs

When a PR changes `apps/editor/src/**`, `site/**`, or workspace demo behavior,
include one of the following:

- an AUB-generated Blueprint/template for the changed surface plus an
  implementation report, or
- a short explanation for why the change is non-visual and does not need an AUB
  report.

## First Surfaces To Cover

1. Demo-mode landing/editor shell.
2. Workspace settings drawer.
3. Component candidate review.
4. PR evidence review panel.

The first committed dogfood surface is the workspace drawer:

- `examples/dogfood/editor-workspace-drawer.aub.template.json`
- `examples/dogfood/editor-workspace-drawer.ui.json`
- `examples/dogfood/editor-workspace-drawer.implementation-report.json`
- `examples/dogfood/aub.ci.json`

Verify it with:

```bash
pnpm ci:verify -- --config examples/dogfood/aub.ci.json --require-reports --require-evidence --min-safety-score 70
```

## Evidence Bar

An AUB self-dogfood report should include:

- node mappings back to the changed source files,
- acceptance evidence for desktop, tablet, and mobile behavior,
- overflow evidence,
- component reuse or code-diff evidence,
- the PR Safety Score comment generated by `pnpm report:comment`.

## Why This Matters

If AUB cannot use AUB to review its own editor changes, the product claim is too
weak. The public proof should be a real PR where AUB catches missing evidence or
confirms an evidence-backed UI change.
